### 分层架构规范

```
┌─────────────────────────────────────┐
│         Frontend (Vue.js)           │
└─────────────────┬───────────────────┘
                  │ HTTP API
┌─────────────────▼───────────────────┐
│      Controller Layer               │  ← API 入口
│  (LynxeController, etc.)            │
└─────────────────┬───────────────────┘
                  │ 调用
┌─────────────────▼───────────────────┐
│       Service Layer                 │  ← 业务逻辑
│  (PlanningCoordinator, etc.)        │
└─────────────────┬───────────────────┘
                  │ 协调
┌─────────────────▼───────────────────┐
│      Executor Layer                 │  ← 执行引擎
│  (DynamicToolPlanExecutor, etc.)    │
└─────────────────┬───────────────────┘
                  │ 使用
┌─────────────────▼───────────────────┐
│       Tool/Agent Layer              │  ← 能力实现
│  (BaseAgent, AbstractBaseTool)      │
└─────────────────┬───────────────────┘
                  │ 持久化
┌─────────────────▼───────────────────┐
│      Repository Layer               │  ← 数据访问
│  (JPA Repositories)                 │
└─────────────────────────────────────┘
```

### 子包组织规范

#### entity/ 包
- `po/`: Persistent Object（持久化对象，对应数据库表）
- `vo/`: Value Object（值对象，业务数据传输）

#### controller/ 包
- 按业务功能划分控制器
- 每个控制器负责特定的 API 端点组

#### service/ 包
- 接口和实现类放在同一目录
- 复杂服务可创建子包组织

#### model/ 包
- `po/`: 数据库实体
- `vo/`: 业务值对象
- `enums/`: 枚举类型
- `converter/`: 类型转换器

```
┌─────────────────────────────────────┐

                  │ 调用
┌─────────────────▼───────────────────┐
│    Composables (组合式函数层)        │

└─────────────────┬───────────────────┘
                  │ 访问
┌─────────────────▼───────────────────┐
│      Stores (状态管理层)             │
│  (Pinia 状态存储)                    │
└─────────────────┬───────────────────┘
                  │ 调用

│       API Layer (API 接口层)         │
│  (HTTP 请求封装)                     │

```

### 目录使用规范

#### api/ 目录
- 所有与后端 API 的交互都封装在此目录
- 使用 TypeScript 接口定义请求和响应类型
- 统一错误处理

#### components/ 目录
- 按功能模块划分子目录
- 组件应该是可复用的
- 复杂组件可拆分为子组件

#### composables/ 目录
- 封装可复用的组合式函数
- 以 `use` 开头命名
- 返回响应式数据和方法

#### stores/ 目录
- 使用 Pinia 管理全局状态
- 每个 store 对应一个业务领域
- 包含 state、getters、actions

#### types/ 目录
- 所有 TypeScript 类型定义
- 按业务模块划分文件
- 导出统一的类型接口

## 代码组织最佳实践

### 1. 单一职责原则
每个包、类、函数应该只有一个改变的理由。

### 2. 依赖方向
上层依赖下层，下层不依赖上层：
- Controller → Service → Repository
- Views → Components → Composables → Stores → API

### 3. 循环依赖避免
- 接口定义在独立的包中
- 使用依赖注入解耦
- 事件机制解耦复杂依赖

### 4. 模块边界清晰
- 每个模块有明确的职责
- 模块间通过定义良好的接口通信
- 避免跨层直接访问
